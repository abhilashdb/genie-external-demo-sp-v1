Why each hop exists

  • Local login — simulates an enterprise app where end-users don't have their own Databricks identity. Any IdP (Okta, Entra ID) could replace this in production; the behavior after auth is identical.
  • User to SP mapping — the trust boundary. Each end-user is deterministically assigned to a dealership-scoped SP. Swapping SPs is how we isolate data between tenants.
  • OAuth M2M token exchange — Databricks doesn't care who the human user is; it authorizes the SP. The backend holds SP credentials (never exposed to the browser) and trades them for short-lived bearer tokens.
  • Row-level security — current_user() inside UC returns the SP's applicationId. The row-filter function maps that UUID to a dealership, so the same SQL query yields different rows for different SPs. Genie produces the SQL without knowing RLS exists; UC enforces.
  • Transparency pane (SSE) — every hop publishes a flow event over the SSE channel bound to the user's session. The pane lets you trace a single request across 4 services in real time.
